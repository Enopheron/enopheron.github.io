The Core Architecture: A Unified Multi-Sphere Data System

Introduction: Why Existing Tools Lead to a Dead End

My journey into personal data management has always been guided by one non-negotiable principle: my data and my knowledge must remain local. This is precisely why proprietary, cloud-based Swiss Army knives like Notion, Tana, or Asana were ruled out immediately. I demanded absolute control over my data, privacy, and complete independence from third-party servers.
I dove deep into the world of local-first tools and explored it:

  • Obsidian: Excellent as a personal knowledge base (PKM), but completely helpless when it comes to managing complex tasks or logging finances.
  • Logseq: A powerful outliner, but bloated with features like Canvas that I simply do not need.
  • Emacs + Org-mode: A legendary environment that I used for a long time, but one that ultimately requires endless maintenance and a mountain of custom "hacks" just to integrate basic personal workflows.

Right before implementing my current system, my setup was highly minimalist: the Helix Editor, a Markdown LSP, and the Zettelkasten methodology. Yet, even here, I hit a brick wall.

The Inter-Sphere Chasm

The fatal flaw of almost all ready-made software and plugins is that they force you into a specific silo. You end up with one application for tasks, another for your knowledge base, and perhaps a third for tracking expenses. A massive chasm opens up between these spheres. They utilize different data types, different file formats, and completely different interfaces.
You cannot naturally link a fleeting thought in your daily journal to a concrete task, and then link that task to a financial transaction without losing the chronology and historical context. You waste your energy maintaining integration scripts and wrapping your brain around the artificial constraints of someone else's plugins.
I realized I didn't need a tool that worked almost the way I wanted. I needed to strip everything back to the absolute basics and understand how information itself is structured at a foundational level.

The Core Philosophy: What is Data, Really?

This deep process of rethinking led me to a fundamental conclusion. Regardless of the domain—be it a task, a log entry, a financial receipt, or a project—every piece of information at its bedrock level is an abstract Core.
This concept was first born in my mind as an architectural agreement before ever being translated into relational SQLite tables.
The Core rests upon one unyielding rule: information has no essence without a title. The title is a mandatory distillation—a single, self-contained thought. If a thought begins to branch out and require a complex hierarchy (such as Markdown H2 or H3 headers within the same note), my system triggers an immediate rule: Stop. This is a new thought. Create a separate Core for it and link it to the current one

-- The foundation of the entire cognitive architecture
CREATE TABLE core (
    id INTEGER PRIMARY KEY,
    sphere TEXT NOT NULL, -- 'todo', 'logs', 'kdb', 'wdb'
    title TEXT NOT NULL,  -- A mandatory, self-contained thought
    content TEXT          -- Optional, flat description
);

By removing rigid technical constraints and reducing all data types to a homogenous Core, I gained absolute freedom. However, to prevent this freedom from turning the database into chaos, a strict architectural agreement and an extension system (Metadata) were built on top of the Core.

The Layered Architecture: Envelopes of Meaning

Instead of separating entities into isolated tables, my architecture acts as a multi-layered atomic structure, where each level wraps and protects the previous one. This encapsulates fluid thinking within strict relational rules.

Based on the system's structural blueprint, data is organized into three nested levels:

Level 1: Base (The Static Nucleus)

At the absolute center sits the Base layer—the indivisible static core of any information. It consists of the Core record itself, bounded seamlessly by Tags and Links. This is the raw data engine. It doesn't care about time, priorities, or projects; it merely holds the core thought and registers its global relationships.

Level 2: Logs (The Thinking Envelope)

The Base layer is completely wrapped by Level 2: Logs, which acts as the environment for active, chronological thinking.

  • It inherits the entire underlying structure of the Base layer (every log entry is inherently a Core object with its own Links and Tags).
  • It injects execution state and real-time metadata via Logs (Meta) (timestamps, problem/question statuses).

This is where the brain reasons, logs history, and processes ongoing context over time.

Level 3: Container (The Virtual Isolation Sphere)

The outermost boundary is Level 3: Container. Crucially, a Container is a purely virtual isolating envelope. It contains no metadata tables of its own. Its sole architectural purpose is to act as a thematic boundary, grouping and isolating specific Core records and Logs within a concrete topic or roadmap.

When you look at the system through your terminal, you aren't just looking at text; you are peeling back these three conceptual layers from the outer Container down to the central Core.

How Metadata Extends the Base

A crucial architectural nuance here is that metadata tables (todo_meta, logs_meta, etc.) extend the entire Base sphere, not just the Core table.
When I introduce a new domain or sphere, I don't reinvent the wheel. It automatically inherits the full capability of the triad: the power to link to any other object in the system, support for explicit and hidden tags, cascade deletions, and seamless integration into the global graph. Metadata tables merely inject domain-specific attributes (like a task status or a log timestamp) onto a universal base skeleton using shared IDs.

The Link Engine: Lazy Searching via Anchors

Links within the links table are not merely pointers to another note's ID. They are a precise historical record of where a thought originated and exactly where it landed.

CREATE TABLE links (
    id INTEGER PRIMARY KEY,
    source_id INTEGER NOT NULL,      -- The origin of the thought
    target_id INTEGER NOT NULL,      -- The destination Core
    target_anchor TEXT NOT NULL,     -- The text anchor within the target content
    FOREIGN KEY (source_id) REFERENCES core(id) ON DELETE CASCADE,
    FOREIGN KEY (target_id) REFERENCES core(id) ON DELETE CASCADE

When I navigate a link, Python orchestrator completely avoids using complex, fragile UUIDs for individual lines of text. Instead, it relies on a smart lazy-search algorithm:

  1. The script isolates the target object using its target_id.
  2. It fetches the target_anchor string and executes a text search strictly within that isolated object.
  3. Priority is given to headers: If the searched keyword appears both in standard paragraph text near the top and inside a Markdown header further down, the algorithm automatically focuses on the header.

This is entirely logical: if we are referencing a specific concept, it will be designated as a sub-header encapsulating that information in 95% of use cases.

The TUI: Visualizing Thoughts as a Git Branch

I interact with the database exclusively through custom Python scripts. They operate in a Read-Only state when browsing, serving as a highly optimized visual filter, navigator, and search engine.
The display interface is heavily inspired by the philosophy of git branch. Because every entity in the system is homogenous and maps back to the Base architecture, their relationships visualize perfectly as a directed causal tree.
In its collapsed state ("the bird's-eye view"), the structure is incredibly clean, showing only the titles of the cores:

▸ Meta
├─▸ D2 – tool for create
└─▸ Portfolio

The triangle (▸) acts as an interactive toggle. I can expand and collapse threads of thought directly in the terminal. When expanded, a Python text-formatting library handles the rich rendering of the Markdown context, turning flat SQLite rows into beautifully formatted terminal prose. It behaves like a standard directory tree or a native git log --graph, where every step down represents a logical continuation of the previous thought.

The Editing Lifecycle: Terminal Editor and Stage Buffer

Because the TUI utilities operate in a Read-Only state, altering data requires a strictly managed pipeline. Direct, unchecked mutation of the database is impossible.
When I position the cursor over an object in the graph and press Enter, the following pipeline initiates:

  1. Snapshot Generation: The script dynamically aggregates data from the core table and its matching _meta table.
  2. Export to a Temporary File: The aggregated data is dumped into a temporary Markdown file.
  3. Metadata Injection: In an elegant UX touch, if the object contains metadata attributes, they are wrapped in curly braces {...} and appended strictly to the end of the primary title string.
  4. Active Editing: The file opens natively inside the Helix Editor, where a Markdown LSP initializes to provide syntax highlighting and autocomplete. I am free to modify the body text or edit the metadata inside the braces.

Automated Error Prevention: The Stage Buffer

The critical engineering check occurs the moment I save the file and exit the editor (:wq). The Python orchestrator intercepts control and subjects the changes to strict validation rules before committing anything to SQLite:

  • Metadata Validation: If I make a typo inside the curly braces and enter an invalid status (e.g., writing todoo instead of the reserved enum value todo), the database will reject the transaction via a CHECK constraint.
  • Referential Integrity Checks (links): The script validates that I haven't altered or deleted header text or text blocks that other independent cores point to via a target_anchor.

If an error or a broken link is discovered, the script halts the database write operation. It safely shifts my modifications and the discovered conflicts into an isolated environment in the Python application's memory—the Stage (mirroring the staging area in Git).
An interactive conflict-resolution screen opens up in the terminal, presenting two paths: either correct the broken anchors and malformed metadata to match system agreements, or abort and roll back the edits entirely. The system physically prevents the database from falling into a partially broken state.

System Linguistics: Explicit, Hidden, and Contextual Tags

Classification within this architecture is split across distinct scopes: note-level tags and inline contextual tags. To prevent entity duplication and maintain human-readable text, tags are separated into three categories:

Explicit Tags

These are written directly within the text of the main title field. They are prefixed with a plus sign. The defining characteristic is that the tag is an organic component of the sentence itself.
Example: Fix the script for auto-rendering and review it with Ivan
If you strip the words script and Ivan, the sentence loses its structural meaning. The parser extracts these constructions and logs them in the tags table behind the scenes, but they remain perfectly readable as natural human speech within the interface.

Hidden Tags (Metadata)

These are tags tied to the note on a global level. They are declared inside the curly braces {...} at the tail of the title during editing. In the primary TUI tree view, they are completely hidden to avoid visual clutter, yet they remain instantly indexable by the search engine.

Contextual Tags (Containers and Logs)

Because complex objects like a Container or a Logs entry contain an internal Markdown layout, tags can be declared at the sub-header level within the content block. They are captured by the parser and registered in the database with their own specific local scope.
Typo Prevention: To completely eliminate spelling errors when creating global tags, a helper script queries the SQLite database, compiles a list of all existing unique tags, and provides an autocomplete menu directly under the cursor inside the Helix Editor.

Conclusion and Future Roadmap

Designing this architecture did not begin with writing lines of code; it began with deep conceptual modeling in the mind. Treating information as an atomic, baseline Core extended via strict relational metadata allowed me to completely erase the artificial boundaries between different areas of life.
Currently, the system fully handles my knowledge base (KDB), personal software projects (Prj), web research bookmarks (WDB), daily logs (Logs), and task management (Todo).
The Immediate Roadmap:

  • The Fin Sphere: Extending the Base triad with a fin_meta table to seamlessly link cohesive life chains: Cognitive log brainstorming a purchase → A task tracking the research → The final financial transaction ledger entry.
  • Portfolio Automation: Merging the project sphere with a script that automatically parses finalized technical documentation and deploys it directly to my public portfolio website.

This setup has proven that the strict boundaries of a relational database and the fluid creative freedom of flat Markdown text are not just compatible—they form a spectacular synergy for human thought.
